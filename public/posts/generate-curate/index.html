Chaos

This class has been chaotic so far. This has probably been frustrating and confusing for some of you.

I try, as a teacher, to only make you frustrated or confused when it is a frustration or confusion that really happens in the workplace. I do this so you can find solutions for it in a safer environment (i.e. the classroom).

This course is about process and strategy. The reason we have process is to create structure and organization, and therefore avoid confusion. The reason we have Agile strategies is so that we're not so structured that we can't adapt. This is how we avoid frustration.

Today we will continue to list pain points.

I expect you'll have thought of some new ones now that you're a few weeks into your new classes.

As a reminder, pain points are things that are boring / repetitive or hard to remember. Stuff associated with process and strategy is usually boring and/or hard to remember, but so is a lot of work associated with development - syntax, running tests, boilerplates, server configuration, sourcing placeholder art, etc. etc. etc.

Luckily for us, computers are really good at doing repetitive stuff and remembering things.

We can automate literally anything a computer can do, creating triggers, sync'ing applications, passing in arguments, and/or chaining together commands.

The list of things computers can do is getting bigger every day. If your pain point involves finding artwork, or sending a text or email, or writing (either English or code), these are all things that can be automated (albeit with some costs associated beyond a certain point).

We can also have computers remember things for us in ways that are the most convenient for us - stored on a server that's accessible to us on all our devices, and/or on a collaborative platform, and/or in a format that's simple and easy to read and integrate with automations, and/or set up for automated reminders (both time- and location-based).

We'll take 10 minutes for you to individually list some of your own "pain points". These can be things that are currently painful, or things you know you probably should do, but aren't currently doing because it's too much work. This can be school stuff, or any tasks outside of school (that maybe you've neglected because you're schooling so hard).

Once you have some pain points listed, we'll have 20 minutes for you to write out the steps involved in each painful task. Take note of anything that might change the next time you do this task. Take note of anything that you'd like to improve about how this task gets done. For each step, it may help you to mark it either (A) for action, or (M) for memory.

Example:

Pain point: boilerplate folder structure
Description: It's hard to remember all the files and folders I should create for a project when I just want to start coding.
Steps:
1. Create a project folder
2. Create subfolders
3. Create boilerplate index.html, /js/script.js, /css/style.css
Note: this is for a static HTML site. Step 3 is slightly different for PHP projects.

Example:

Pain point: remembering due dates
Description: I can manually put due dates for assignments in my google calendar, but they're scattered all over, and it doesn't reflect how much time each assignment will take.
Steps:
1. Take a note in class
2. Manually add to Google Calendar
Note: I wish I could see how much time each task will take, and how many marks each task is worth. That way I'd know what to prioritize.

After listing out your pain points, we'll get into groups based on solutions you'd like to explore for these pain points.

  • Documentation (collaborative 'wiki'-style note-keeping in plain text/markdown)
  • Workflow automation (Github Actions, chaining together tools, testing, linting, etc.)
  • Boilerplating (creating reproducible "start points" for coding projects)
  • Task automation (having your own computer perform simple, repetitive tasks via shell scripting, Python, or similar)
  • Something else - anything that at least 5 people would like to explore together (that can ultimately be captured in Github).
